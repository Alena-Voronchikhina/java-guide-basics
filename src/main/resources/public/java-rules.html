Java Hard Constraints (Enforced)

❗️ These are non-negotiable Java platform rules. If design conflicts with them, redesign the code.

Source and Compilation

// One public top-level class per file public class Foo { public static void main(String[] args) { System.out.println("Hello"); } } // File name must be Foo.java // Java is case-sensitive // Statements end with semicolons // Java is statically typed (var is compile-time inference only)

Primitive Types (Fixed Sizes)

byte // 8-bit short // 16-bit int // 32-bit long // 64-bit char // 16-bit UTF-16 code unit float // 32-bit IEEE 754 double // 64-bit IEEE 754 // No unsigned primitives except char // Sizes are platform-independent

Object Model Rules

// Objects are heap-allocated // Variables hold references, not raw object values // Java passes arguments by value // For objects: value copied is the reference // No pointer arithmetic // No user-visible stack allocation for objects

Packages, Class Loading, and Entry Point

package a.b.c; // maps to a/b/c/ public class App { public static void main(String[] args) { System.out.println("Entry point"); } } // Classpath resolution uses ordered lookup // Class shadowing is possible // Reverse-domain package names are ecosystem standard

Runtime and JVM Guarantees

// Garbage collection is mandatory // No deterministic destruction (no RAII) // Use try-with-resources for cleanup // Checked exceptions are enforced Throwable |- Exception |- Error // Stack-based JVM model is fixed // System.out and System.err are always present // finalize() is effectively obsolete

Design Rule

If an abstraction conflicts with Java constraints: 1) Do not fight Java 2) Redesign the abstraction 3) Assume the constraint is permanent
